## 3. Ustalenia — APLIKACJA (Strapi API + Angular)

### Wysokie (P1)

- **A1 ✅ JWT w `localStorage`** — `frontend/src/app/core/services/auth.service.ts:108,126,148`. Token dostępny dla JS → wektor kradzieży przy XSS. Znane/„zaakceptowane" w audycie 2026-05-04 (B5). Rekomendacja: cookie `HttpOnly; Secure; SameSite` ustawiane przez serwer + ochrona CSRF; albo świadoma akceptacja ryzyka na soft-launch (wpis do decision log).
- **A2 ✅ Sekret webhooka newslettera porównywany `===`** — `apps/api/src/api/newsletter/controllers/newsletter.ts:243` (`headerSecret === expected || authorization === 'Bearer ' + expected`). Podatność na timing-attack (niska w praktyce) + brak HMAC podpisu payloadu; poza prod sekret bywa pomijany. Rekomendacja: `crypto.timingSafeEqual`, wymóg sekretu też poza prod, rozważyć HMAC.
- **A3 ⚠️ Brak idempotency-key przy Stripe Checkout** — `apps/api/src/api/checkout/controllers/checkout.ts` (tworzenie sesji przez `fetch`). Retry/double-click → duplikaty sesji. Rekomendacja: nagłówek `Idempotency-Key`.
- **A4 ⚠️ Zaufanie do `X-Forwarded-For` w rate-limit** — `apps/api/src/middlewares/rate-limit.ts`. Jeśli Caddy nie nadpisuje nagłówka, możliwy spoofing IP i obejście limitów. Rekomendacja: potwierdzić, że reverse-proxy nadpisuje XFF, a `trust proxy` jest włączone tylko za zaufanym proxy.

### Średnie (P2)

- **A5 ✅ Ciche połykanie błędów Redis** — `rate-limit.ts` (`client.on('error', () => undefined)`). Degradacja niewidoczna, brak alertu. Dodatkowo `failOpen` domyślnie `true` poza prod.
- **A6 ✅ Cart: brak persystencji + kwoty na `float`** — `libs/frontend/cart/src/lib/*`. Ryzyko utraty koszyka i błędów zaokrągleń (IEEE754). Rekomendacja: liczyć w groszach (int), persystencja w storage.
- **A7 ✅ Dług typów** — ~71 `as any` w API, ~29 we froncie; `noUnusedLocals:false` w `tsconfig.base.json` (martwy kod niewykrywany). Liczby przybliżone.
- **A8 ⚠️ Turnstile bez timeoutu na `fetch`** — `apps/api/src/utils/turnstile.ts`; ryzyko zawieszenia żądania przy wolnym Cloudflare.
- **A9 ✅ Walidacja e-mail luźnym regexem** — `contact.ts`; drobne, warto zaostrzyć.

### Testy aplikacji

- **A10 ✅** API ~7% pokrycia (≈21 plików testów); **brak testów** dla contact / newsletter / checkout / turnstile / order. Front ~75–85% (dobrze). E2E: 5 plików, głównie happy-path — brak scenariuszy negatywnych (błąd webhooka Stripe, rate-limit, błąd publikacji).

## 4. Ustalenia — PLUGIN AICO

### Wysokie (P1)

- **C1 ✅ Wyścig przy przejęciu wygasłej blokady** — `services/runtime-locks.ts:70-81`. `lock_key` ma `unique:true` (schema), więc wyścig *tworzenia* jest obsłużony (drugi `create` rzuca → `held`). Ale ścieżka przejęcia wygasłego/released locka to **read-then-update bez atomowości**: dwie instancje widzą wygasły lock i obie robią `update` na tym samym rekordzie → obie dostają `acquired:true`. TTL=55s < tick=60s, a `release` ustawia `released` w `finally` → przy **multi-instance** kolejny tick może wystartować równolegle na obu węzłach. Skutek: podwójna generacja/publikacja/**wydatek na reklamy**. Obecnie deploy jednoinstancyjny (niskie realne ryzyko *teraz*), ale latentny błąd uniemożliwiający bezpieczne skalowanie poziome. Rekomendacja: warunkowy atomowy `UPDATE ... WHERE (status='released' OR expires_at<=now)` ze sprawdzeniem liczby zmienionych wierszy, lub `pg_advisory_lock`.
- **C2 ✅ Nieatomowy dzienny cap budżetu reklam** — `services/ads-budget-ledger.ts` (`reserveActivation`, ~l.83-174). `unique_key` (`unique:true`) chroni idempotencję pojedynczego planu, ale kontrola „suma aktywnych ledgerów < cap" to read-sum-then-create. **Różne plany** rezerwujące równolegle mogą wspólnie przekroczyć `daily_ads_budget_pln`. W połączeniu z C1 (dwa ticki) ryzyko realnego przekroczenia limitu wydatków. Rekomendacja: serializacja rezerwacji (advisory lock per `platform+day`) albo egzekwowanie capu warunkiem bazodanowym.
- **C3 ⚠️ Słaba walidacja wejścia w admin API** — `services/workflows.ts` (create/update), wyrażenia cron, limity (`retry_max`, `daily_token_limit`, `timezone`). Typy TS bez walidacji runtime; nieprawidłowy cron/limit może przejść. Rekomendacja: `zod` na wejściu (część walidacji już istnieje — potwierdzić zakres).
- **C4 ✅ Powierzchnia prompt-injection** — `orchestrator.ts:2529` (`renderPrompt` prostym `replace`) + temat/treść użytkownika trafiają do LLM. Łagodzące: dostęp tylko dla adminów (RBAC), output walidowany schematem (`validateHoroscopePayload`, `orchestrator.ts:2535`). Rekomendacja: whitelist placeholderów, limit tokenów *wejścia*.
- **C5 ✅ Brak rate-limitu na trasach admina pluginu** — `routes/admin/index.ts` (np. `/workflows/:id/run-now`, `run`, `backfill`). Uwierzytelniony admin może wielokrotnie odpalać kosztowne operacje (tokeny LLM, media, ads). Rekomendacja: throttle per-user/operacja.

### Średnie (P2)

- **C6 ✅ Audyt „fail-skip" domyślnie** — `utils/audit-trail.ts`; przy awarii audytu zdarzenie pomijane (chyba że `AICO_AUDIT_TRAIL_STRICT=true`). Rekomendacja: retry/kolejka, by ślad audytowy był niezawodny.
- **C7 ⚠️ Granice dzienne budżetu w UTC vs timezone workflow** — `services/autonomy-policy.ts` (`startOfTodayIso` liczy dzień w UTC). Reset capu o złej porze lokalnej (Europe/Warsaw). Rekomendacja: liczyć dobę wg timezone workflow.
- **C8 ⚠️ Pliki tymczasowe media** — `services/media-generator.ts` (`auto_${Date.now()}.webp` w `public/uploads/tmp`); niejasne sprzątanie i brak kontroli miejsca na dysku.
- **C9 ✅ `lock_key` typu `uid`** — `content-types/runtime-lock/schema.json`. `uid` potrafi slugifikować wartości (klucze zawierają `:`, np. `orchestrator.tick`); nietypowe dla klucza programistycznego, choć `unique` działa. Rozważyć `string` + indeks unikalny.
- **C10 ✅ Krytyczny dług testowy** — ~4–10 plików testów na 91/30,5k LOC; **brak testów dla `orchestrator.tick` (2706 LOC) i `social-publisher` (2333 LOC)** — najwyższe ryzyko bez pokrycia.
- **C11 ✅ ~44 markery** `as any`/`TODO`/`eslint-disable` w pluginie (umiarkowanie jak na 30k LOC).
- **C12 ⚠️ Brak polityki retencji** dla `audit-event` / `run-log` / śladów → nieograniczony wzrost danych.

### Sprostowania wobec wstępnej analizy (obniżona waga — ważne dla rzetelności)

- **SSRF/URL reklam jest faktycznie solidny** — `ads-agent.ts:222-257`: wymóg `https`, blok userinfo, **allowlista hostów i prefiksów ścieżek**, blok fragmentu, blok kluczy query „tokenopodobnych". `new URL` normalizuje IDN→punycode. Pozostaje **drobiazg (P3)**: nie sprawdza *wartości* query. Wcześniejsza ocena „wysokie SSRF" — **nietrafiona**.
- **Polityka autonomii na poziomie ticka fail-CLOSED** — `orchestrator.ts:416-435`: przy błędzie/niedostępności polityki tick jest **blokowany** (nie publikuje). Wcześniejsze „fail-open" dotyczyło per-action `evaluate` (do potwierdzenia) — globalna bramka jest bezpieczna.
- **Sekrety/redakcja OK** — tokeny szyfrowane (Strapi encryption), błędy providerów redagowane (`open-router.ts`), surowe ślady LLM domyślnie nieprzechowywane. To mocne strony, nie ryzyka.

## 7. Weryfikacja / jak testować naprawy

- **Współbieżność (C1/C2):** test integracyjny odpalający równolegle N `acquire(klucz)` / `reserveActivation(plan)` na wspólnej bazie i asercja, że jest dokładnie jeden lock i suma rezerwacji ≤ cap (vitest + testowy Strapi, wzorzec z `runtime.test.ts`).
- **AICO end-to-end:** `npm run ops:predeploy:staging` z realnym `.env` → preflight `passed`; `/autonomy/production-readiness` i `/audit/preflight` = GO; smoke reklam w trybie `controlled` + test stop-loss.
- **Aplikacja:** testy jednostkowe ścieżek (contact/newsletter/checkout) + E2E negatywne; ręczny test live Stripe (podpis webhooka) i GA4 DebugView.
- **CI/supply-chain:** PR sprawdzający, że Trivy CRITICAL/HIGH **blokuje** pipeline; `gitleaks detect` pełnej historii = 0; build z przypiętymi digestami.
